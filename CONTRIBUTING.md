# Contributing to Storage Options Documentation

Thank you for your interest in contributing to the Storage Options documentation! This document provides guidelines and instructions for contributing.

## How to Contribute

### Reporting Issues

If you find errors, unclear sections, or missing information:

1. **Check existing issues** to see if the problem has already been reported
2. **Create a new issue** with:
   - Clear description of the problem
   - Steps to reproduce (if applicable)
   - Expected vs actual behavior
   - OpenShift/ARO/ROSA version (if relevant)
   - Screenshots or error messages (if applicable)

### Suggesting Improvements

We welcome suggestions for:
- Additional storage options or configurations
- Better explanations or examples
- New use cases or scenarios
- Performance optimization tips
- Security best practices

### Submitting Changes

1. **Fork the repository**
2. **Create a feature branch** (`git checkout -b feature/amazing-improvement`)
3. **Make your changes** following our style guidelines
4. **Test your changes** if applicable
5. **Commit your changes** with clear, descriptive commit messages
6. **Push to your branch** (`git push origin feature/amazing-improvement`)
7. **Open a Pull Request** with a clear description of changes

## Documentation Style Guidelines

### Markdown Formatting

- Use proper heading hierarchy (H1 for title, H2 for main sections, H3 for subsections)
- Use code blocks with language identifiers (````yaml`, `bash`, etc.)
- Use tables for comparisons
- Include examples with clear explanations

### Code Examples

- **Always include comments** explaining what each section does
- **Use placeholder values** clearly marked (e.g., `<your-cluster-name>`)
- **Include expected output** when showing commands
- **Test all code examples** before submitting

### Structure

- Start with an overview/introduction
- Include prerequisites section
- Provide step-by-step instructions
- Include validation/testing steps
- Add troubleshooting section when relevant

### Writing Style

- **Be clear and concise**
- **Use active voice** when possible
- **Explain technical terms** on first use
- **Include context** for commands and configurations
- **Provide real-world examples** when possible

## Content Guidelines

### What to Include

- ✅ Clear prerequisites and version compatibility
- ✅ Step-by-step instructions
- ✅ Complete, working code examples
- ✅ Security considerations
- ✅ Troubleshooting tips
- ✅ Validation checklists
- ✅ Real-world use cases

### What to Avoid

- ❌ Platform-specific assumptions without noting them
- ❌ Outdated or deprecated methods
- ❌ Incomplete code examples
- ❌ Missing prerequisites
- ❌ Hardcoded credentials or secrets
- ❌ Vague or ambiguous instructions

## Review Process

1. **Automated checks**: PRs are checked for formatting and basic validation
2. **Community review**: Community members may review and comment
3. **Maintainer review**: Maintainers review for accuracy and completeness
4. **Merge**: Once approved, changes are merged to main branch

## Areas Needing Contribution

We particularly welcome contributions in these areas:

- **Additional storage vendors** and their OpenShift integration
- **Performance tuning** guides and benchmarks
- **Disaster recovery** procedures and runbooks
- **Cost optimization** strategies
- **Multi-cloud** storage scenarios
- **Edge computing** storage solutions
- **Security hardening** guides
- **Troubleshooting** guides for common issues

## Questions?

If you have questions about contributing:
- Open an issue with the `question` label
- Check existing documentation first
- Review similar contributions for examples

Thank you for helping improve this documentation!
